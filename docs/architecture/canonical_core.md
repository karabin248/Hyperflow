Status: canonical
Confidence: high

## Purpose

This document defines the **current canonical core** of Hyperflow at the MVP Runtime Baseline stage.

It answers four questions clearly:

1. What Hyperflow is now
2. What the core layers are
3. What belongs inside the baseline
4. What does **not** belong inside the baseline

## Canonical stage

At the current stage, Hyperflow is defined as:

- **MVP Core**
- **Runtime Shell**
- **Minimal Orchestration Layer**

It is **not yet** defined as:

- full agent runtime
- platform layer

At the MVP Runtime Baseline stage, Hyperflow is best understood as a layered runtime system:

- **Hyperflow** = system shell / architecture frame
- **MPS** = state-policy / mode-control layer
- **EDDE** = process execution contract
- **Observer / thresholds** = guardrail layer
- **Fragment / structured output** = output object layer

- a stable runtime shell

- guardrails against drift
- structured output behavior

## Canonical cycle

The canonical cycle is `🌈💎🔥🧠🔀⚡`.

Meaning:

- `🌈` = perceive
- `💎` = extract essence / core
- `🔥` = set direction
- `🧠` = synthesize
- `🔀` = generate / compare options
- `⚡` = choose

Canonical logic:

- synthesis happens before option comparison
- option comparison happens before choice
- choice is the final commitment step

Therefore:

- `🌈💎🔥🧠🔀⚡` = canonical
- `🌈💎🔥🧠⚡🔀` = legacy-only, non-canonical

## Minimal stable core

The minimal stable core consists of five layers.

### 1. Input Control Layer
Responsible for emoji/control parsing, input interpretation, and command/control mapping.

### 2. State Control Layer
Responsible for MPS state-policy behavior, mode selection, and transition policy.

### 3. Process Runtime Layer
Responsible for EDDE execution flow, runtime phase sequencing, and deterministic runtime behavior.

### 4. Guardrail Layer
Responsible for observer logic, thresholds, rollback-safe boundaries, and drift protection.

### 5. Output Object Layer
Responsible for structured output, final result shaping, and fragment-compatible output contract.
